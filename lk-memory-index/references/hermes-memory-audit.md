# Hermes 持久存储审计与压缩流程

> 对比 `memory` 和 `user profile` 两个持久存储区，去重、压缩、更新知识库的全流程。
> 参考会话：2026-06-12 记忆结构重构（15条→3条，1,196→289 chars）。

## 触发条件

- 用户说"查询记忆规则有没有不需要的"或类似指令
- user profile 占用 >75%（接近 1,375 chars 上限）
- 发现同一规则既在 memory 又在 user profile 中
- 知识库 `个人画像.md` 与实际 memory 状态不一致

## 审计步骤

### 1. 读取当前状态

```python
# 读取两个存储区的全部内容
# 使用 memory tool 的 add action 来读取（add 后自动显示全部 entries）
# 然后 remove 掉临时的 placeholder
```

读完后评估：
- memory 有多少条？占用百分比？
- user profile 有多少条？占用百分比？
- 两条存储区之间是否有重复内容？

### 2. 跨存储区去重

同一规则出现在 memory + user profile 中时：
- **谁更合适？** user profile 优先于 memory（profile 的规则会强注入，memory 次之）
- **移除**：删掉 memory 中的副本，保留 user profile 中的完整版
- 例外：若 skill 中有完整细节，profile 只留指针，skill 保留完整版

### 3. 压缩策略

| 当前内容类型 | 操作 | 去向 |
|-------------|------|------|
| 长技术细节（>150 chars） | 移至 skill | skill + profile 留指针 |
| 重复规则（跨存储区） | 去重 | 保留更合适的那份 |
| 硬规则（不记生活/删除回收站/注释箭头） | 用户不要则删除 | — |
| 身份信息 | 压缩到影响每次对话判断的最小集 | profile |
| 交互偏好 | 压缩为紧凑格式（`\|` 分隔） | profile |
| 归档格式详情 | 移入已有 skill | profile 留指针 |

### 4. 压缩后写入

将新内容按分类写入 user profile：

```
entry 1: [身份] — 影响每轮判断的核心信息（OS/模型/多profile）
entry 2: [交互+指针] — 偏好 + skill 引用
entry 3: [硬规则] — 最高优先级规则（如 MiMo 先调）
```

### 5. 知识库同步

更新 `wiki/配置/个人画像.md`：
- 模型名称（如有变更）
- 记忆管理章节（§5 存储结构/规则/定时）
- 删除不存在的文件引用（如记忆备份文件）
- 更新最后更新日期

更新 `wiki/配置/AI 检索索引.md` 中指向记忆管理的链接（修复死链）。
更新 `wiki/配置/Agent环境迁移清单.md` 中的记忆备份说明。

### 6. 验证

重新读取两个存储区，确认：
- 占用率显著下降
- 无重复条目
- 重要规则未丢失
- 知识库文件指向正确
